Reference · cesium design system

Design system

Cesium provides a small vocabulary of HTML classes for producing beautiful, self-contained artifact pages — plans, reviews, comparisons, reports, and explainers. Every artifact is a single HTML file that requires no external resources: fonts, colors, and layout are fully inlined. This reference page demonstrates every available class so an agent can internalize the design language before writing a complex artifact.

02 · typography

02Typography & Headings

Use when you need to establish visual hierarchy. The display heading anchors the page; section headings break the document into scannable chunks.

Display heading — the page title

Section heading — a major division

Body copy is set in the system sans-serif at 1rem/1.6. Use sparingly; prefer short paragraphs with plenty of whitespace. Inline code renders in the monospace stack at 0.9em.

Long-form explanatory text sits comfortably at this measure. Keep paragraphs to 3–5 sentences so readers can skim the first sentence of each block.

04 · cards

04Cards

Use .card for any discrete, bordered surface — a finding, a recommendation, a team member bio, or a feature description. Cards sit in a CSS grid for multi-column layouts.

finding 01

Session timeout too short

Users are logged out mid-task. Extend the idle timeout from 15 to 30 minutes and add a one-minute warning banner.

finding 02

No mobile breakpoint

The dashboard overflows on viewports narrower than 768 px. A responsive grid reflow fixes the majority of complaints.

finding 03

Search latency p95 = 4.2 s

Full-text search scans the entire corpus on every keystroke. Add a debounce of 300 ms and an n-gram index to bring p95 under 400 ms.

05 · tldr

05TLDR

Use exactly one .tldr block per document, positioned near the top. It is the executive summary — three to five sentences that give a busy reader everything they need without reading the rest of the document.

The authentication refactor is ready to ship. We replaced the legacy session cookie approach with short-lived JWTs (15 min access, 7 day refresh) backed by a Redis token store. All 142 existing auth tests pass; three new integration tests cover the refresh flow. The breaking change is limited to the /api/v1/auth namespace — no front-end changes required.

06 · callouts

06Callouts

Use callouts to surface information that needs to be noticed but doesn't belong in running prose. Three flavors: .callout.note for helpful context, .callout.warn for things that can go wrong, and .callout.risk for blocking or high-severity concerns.

Note: The refresh token rotation strategy requires a Redis instance with persistence enabled — an in-memory-only instance will cause silent logouts on server restart.
Warning: Deploying this change to production before migrating the existing session cookies will log out all active users simultaneously.
Risk: The token revocation list is stored only in Redis; a cache flush between access-token expiry and refresh will allow briefly-revoked tokens to remain valid.

07 · code

07Code blocks

Use .code for multi-line code samples. Highlight tokens with nested spans: .kw for keywords, .str for string literals, .cm for comments, and .fn for function names. Inline code elements in body copy use the browser default monospace style.

// Refresh an access token using the stored refresh token. export async function refreshAccessToken( refreshToken: string, ): Promise<{ accessToken: string }> { const payload = await verifyRefreshToken(refreshToken); if (!payload) { throw new Error("invalid or expired refresh token"); } const accessToken = await signAccessToken({ sub: payload.sub, scope: payload.scope, }); return { accessToken }; }

Inline usage: press ⌘ S to save, or call refreshAccessToken(token) directly from the REPL.

08 · timeline

08Timeline

Use .timeline for ordered milestones, release schedules, or onboarding sequences. Each <li> gets a dot and a vertical connector automatically via CSS.

  • Week 1 — Audit & planning

    Review existing auth code, identify all call sites, write migration checklist. Deliverable: annotated codebase + go/no-go criteria.

  • Week 2–3 — Implementation

    Build JWT signing, Redis token store, refresh rotation, and revocation list. Feature-flagged behind AUTH_V2.

  • Week 4 — Hardening & rollout

    Load test at 3× peak traffic, canary 10% of prod, monitor error rates for 48 h, then full cutover.

10 · compare-table

10Compare table

Use .compare-table when presenting a decision matrix or feature comparison across two or more options. Keep columns to three or fewer for readability on narrow viewports.

Approach Complexity Operational cost
Session cookies + DB Low — well-understood pattern, many libraries Requires sticky sessions or a shared session store
JWT (stateless) Medium — signing, rotation, and revocation add surface area No server-side state; tokens carry their own claims
JWT + Redis revocation Medium-high — Redis dependency, invalidation logic Adds Redis infra; enables immediate revocation unlike pure JWT

11 · risk-table

11Risk table

Use .risk-table for structured risk registers — likelihood, impact, and mitigation. Use .pill chips inside cells to tag severity levels at a glance.

Risk Likelihood · Impact Mitigation
Redis unavailable at token refresh time Medium · High Circuit-break to read-only mode; allow access tokens to coast until Redis recovers (max 15 min gap).
Private key leaked in environment variable Low · Critical Store key in a secrets manager (Vault / AWS Secrets Manager); rotate on any suspected exposure; never log the key.
Clock skew causes spurious token rejection Medium · Medium Allow a 30-second leeway window in the JWT library; add NTP monitoring to all nodes.
